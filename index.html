Clean Code

Escrevendo Código Python Legível e Manutenível

Introdução


Clean code se refere a um estilo de programação que se concentra em tornar o código fonte mais legível, compreensível e fácil de manter

É um conjunto de práticas e diretrizes que promovem uma estrutura clara, nomes descritivos, funções concisas e outras abordagens que ajudam os desenvolvedores a criar software de alta qualidade

Benefícios do clean code:


  1. Legibilidade aprimorada

  2. Facilidade de manutenção

  3. Redução de bugs e erros

  4. Colaboração eficiente em equipes

Legibilidade aprimorada:


  • O código com uma estrutura clara e nomes descritivos facilita a compreensão do que está acontecendo em cada parte do programa.
  • Outros desenvolvedores (incluindo você mesmo no futuro) poderão entender mais facilmente o código e fazer modificações quando necessário.
  • A legibilidade também ajuda na depuração e na identificação rápida de erros.

Facilidade de manutenção:


  • O código limpo é mais fácil de manter. Quando você precisa fazer alterações ou corrigir erros, é mais rápido localizar a parte relevante do código e realizar as modificações necessárias.
  • Menos tempo é gasto na decodificação e na compreensão de trechos de código confusos ou mal estruturados.
  • Isso resulta em um processo de manutenção mais eficiente e menos propenso a introduzir novos bugs.

Redução de bugs e erros:


  • O código limpo tende a ter menos bugs. Quando você segue boas práticas de clean code, é menos provável que cometa erros e introduza bugs.
  • A clareza do código também ajuda a identificar e corrigir erros mais rapidamente durante o processo de depuração.
  • Além disso, a aplicação de técnicas como a separação de responsabilidades e a modularização do código contribui para um código mais robusto e menos propenso a erros.

Colaboração eficiente em equipes:


  • O código limpo é mais fácil de compartilhar e colaborar com outros membros da equipe. Cada pessoa pode entender o código rapidamente e contribuir de forma mais eficaz para o projeto.
  • Uma base de código limpa promove um ambiente de trabalho harmonioso e facilita a manutenção do código ao longo do tempo.
  • Além disso, ao seguir as práticas de clean code, você estará criando um padrão de qualidade que pode ser adotado por toda a equipe, garantindo consistência e uniformidade no código.

Nomes descritivos de variáveis, funções e classes:


  • Use nomes que expressem claramente o propósito e a função das suas variáveis, funções e classes.
  • Evite abreviações ou nomes genéricos que possam levar a ambiguidades ou dificuldades de entendimento.
  • Nomes descritivos tornam o código mais legível e facilitam a compreensão do que cada elemento faz.
  • Mantenha a nomenclatura consistente para evitar confusão e variação desnecessária no código.

Princípio do Aberto/Fechado (OCP)

“Incorporate new features by extending the system, not by making modifications (to it)”
Robert C. Martin

Os objetos devem estar abertos para extensão, mas fechados para modificação. Deve ser possível ampliar a funcionalidade fornecida por um objeto (por exemplo, uma classe) sem alterar seus contratos internos. Um objeto pode permitir isso quando é projetado para ser estendido de forma limpa.

Princípio do Aberto/Fechado (OCP)

⚠️
                            
from dataclasses import dataclass


@dataclass
class Response:
    status: int
    content_type: str
    body: str


class View:
    def get(self, request) -> Response:
        return Response(
            status=200,
            content_type='text/plain',
            body="Welcome to my web site"
        )


class TemplateView(View):
    def get(self, request) -> Response:
        with open("index.html") as fd:
            return Response(
                status=200,
                content_type='text/html',
                body=fd.read()
            )

from dataclasses import dataclass, field
from typing import Protocol


@dataclass
class Response:
    """An HTTP response"""

    status: int
    content_type: str
    body: str
    headers: dict = field(default_factory=dict)


class View:
    """A simple view that returns plain text responses"""

    content_type = "text/plain"

    def render_body(self) -> str:
        """Render the message body of the response"""
        return "Welcome to my web site"

    def get(self, request) -> Response:
        """Handle a GET request and return a message in the response"""
        return Response(
            status=200,
            content_type=self.content_type,
            body=self.render_body()
        )
                            
                        
                            
class TemplateRenderMixin:
    """A mixin class for views that render HTML documents using a template file

    Not to be used by itself!
    """
    template_file: str = ""

    def render_body(self) -> str:
        """Render the message body as HTML"""
        if not self.template_file:
            raise ValueError("The path to a template file must be given.")

        with open(self.template_file) as fd:
            return fd.read()


class ContentLengthMixin:
    """A mixin class for views that injects a Content-Length header in the
    response

    Not to be used by itself!
    """

    def get(self, request) -> Response:
        """Introspect and amend the response to inject the new header"""
        response = super().get(request)  # type: ignore
        response.headers['Content-Length'] = len(response.body)
        return response


class TemplateView(TemplateRenderMixin, ContentLengthMixin, View):
    """A view that returns HTML responses based on a template file."""

    content_type = "text/html"
    template_file = "index.html"
